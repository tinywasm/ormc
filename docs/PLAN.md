# PLAN — `tinywasm/ormc`: dejar de depender de `tinywasm/ddlc`

Pieza de una migración mayor documentada en
[`DDLC_DEPENDENCY_PROPOSAL.md`](https://github.com/tinywasm/app-releases/blob/main/docs/DDLC_DEPENDENCY_PROPOSAL.md)
(repo `tinywasm/app-releases`). **Autocontenido, en español. Solo tienes este repo**
(`github.com/tinywasm/ormc`) — no necesitas leer el link para ejecutar este plan, pero
ahí está el razonamiento completo si algo no cuadra.

## 0. Contexto — por qué este cambio

`tinywasm/ddlc` iba a ser **solo la herramienta** de generación de DDL (el `tui` de
migraciones), pero hoy además define tres piezas de **contrato de runtime**:

- `FieldExt` — un tipo de datos: metadata de foreign key (`Ref`, `RefColumn`, `OnDelete`)
- `TopologicalSort` — un algoritmo de ordenamiento de tablas por dependencia FK
- `Exporter` — una interfaz (`ExportDDL(models) (string, error)`)

Por eso cualquier modelo de usuario con una foreign key terminaba arrastrando "la
herramienta" dentro del código generado (y potencialmente al binario WASM del frontend).
Ya se ejecutaron dos pasos previos de esta migración:

- `FieldExt` **ya se movió** a `github.com/tinywasm/model` (idéntico shape, `model.FieldExt`),
  publicado en `model@v0.0.16`.
- `TopologicalSort` **ya se movió** a `github.com/tinywasm/ddl` (`ddl.TopologicalSort`),
  publicado en `ddl@v0.0.4`.

Este plan es el **tercer paso**: `ormc` es el generador de código que hoy emite
`ddlc.FieldExt` en el código de cada modelo con FK, y además acepta un `ddlc.Exporter`
en `ExportSQL`. Ambos dejan de ser necesarios: `model.FieldExt` ya existe (`ormc` ya
importa `tinywasm/model` incondicionalmente en todo archivo generado), y `Exporter` es
una interfaz de un solo método — en Go, quien la **consume** la define, no quien la
implementa (tipado estructural: `sqlt.NewCompiler()`/`postgres.NewCompiler()` ya
implementan el método `ExportDDL`, no necesitan importar nada para satisfacer una
interfaz nueva definida en `ormc`).

**No toques nada relacionado con `github.com/tinywasm/orm`** (el import `"github.com/tinywasm/orm"`
que ves en `generate.go` para `*orm.QB`, `ReadOne`/`ReadAll`). Es una migración distinta,
fuera de alcance de este plan. Solo tocas lo que menciona explícitamente cada etapa.

## 1. Qué se hace

- `generate.go`: el código generado (`*_orm.go`) emite `func (m *X) SchemaExt() []model.FieldExt`
  en vez de `[]ddlc.FieldExt` — y ya no necesita un import condicional de `ddlc` (el import
  de `model` ya es incondicional en todo archivo generado).
- `generate.go`: `ExportSQL` deja de aceptar `ddlc.Exporter` — define su propio tipo
  `Exporter` local (mismo método, mismo shape) y lo acepta en su lugar.
- `go.mod` (raíz) y `tests/go.mod`: **fuera** `github.com/tinywasm/ddlc` por completo.
- Los tests que verifican el string generado (`"[]ddlc.FieldExt"`) pasan a esperar
  `"[]model.FieldExt"`.
- El fixture generado `tests/models_orm.go` se **regenera** (no se edita a mano — ver
  §3.4) para reflejar el nuevo output.
- `docs/ARCHITECTURE.md`: dos líneas que describen el contrato viejo se actualizan.

### 3.5 `generate.go` — `Exporter` local + `ExportSQL`

Justo antes de `ExportSQL` (o en cualquier punto del archivo antes de su uso — la posición
exacta no importa, Go no requiere orden de declaración), añade:

```go
// Exporter is implemented by SQL adapter compilers (sqlt, postgres) that can render a
// full schema export. Defined here — not imported from ddlc — per Go convention: the
// consumer of a single-method interface owns the interface, not the implementer.
type Exporter interface {
	ExportDDL(models []model.Model) (string, error)
}
```

Y cambia la firma de `ExportSQL` (el cuerpo de la función no cambia, solo el tipo del
parámetro y el comentario):

```go
// ExportSQL scans the directory for models and returns the full DDL.
// Requires an injected Exporter (e.g. from sqlt or postgres).
func (g *Generator) ExportSQL(root string, exporter Exporter) (string, error) {
```

(antes: `exporter ddlc.Exporter`, comentario decía `injected ddlc.Exporter`)

> No hace falta tocar ningún caller de `ExportSQL` fuera de este repo (p.ej.
