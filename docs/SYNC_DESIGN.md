# Design Rationale & Honest Trade-offs ‚Äî Tool-driven Dev Schema Sync

> Honest assessment of the architecture behind `db.Sync`/`db.SyncSchema` and the `tinywasm/app`
> hot-schema-sync loop. Not marketing ‚Äî the goal is to know exactly what we're buying and what we're
> deferring. See [DB_SYNC diagram](diagrams/DB_SYNC.md) for the flow and `docs/PLAN.md` for the
> contract.

## The approach in one paragraph

While developing, the dev tool (`tinywasm/app`) watches `model.go` files. On each change it
regenerates the ORM code (`ormc`, AST-based) and **applies an additive schema reconcile directly to
the dev database in real time** ‚Äî add tables/columns, rename via an `old_name` hint, drop only empty
columns. There are **no migration files**: the Go struct is the desired state, git is the model's
history. The **sync code is not in the user's project** ‚Äî the tool owns the DB connection (from
`.env`) and applies the schema through an injected, DB-blind `SchemaSyncer`. Engine selection is a
`database/sql`-style registry (`orm.Open("postgres://‚Ä¶")`).

## This is not a novel idea ‚Äî prior art

The core pattern (**"push the model straight into the dev DB, no migration files"**) is well-trodden.
Being honest about that matters:

| Tool | Equivalent feature | Same idea? |
|------|--------------------|-----------|
| **Prisma** (Node) | `prisma db push` | **Yes ‚Äî almost identical.** Push the schema to the dev DB, no migration history. Prisma explicitly markets it for *prototyping* and pairs it with `prisma migrate` (versioned) for real work. |
| **GORM** (Go) | `AutoMigrate` | **Yes.** Additive auto-migrate from structs (adds, never deletes). Widely used in dev, widely cautioned against in prod. |
| **Hibernate** (Java) | `hbm2ddl.auto=update` | **Yes.** Additive auto-update from entities. The canonical "never in prod" footgun. |
| **Ent + Atlas** (Go) | declarative + versioned | Codegen like `ormc`, but Atlas computes a real **diff** and can emit **versioned files** for prod. The mature version of this idea. |
| **Atlas** (ariga) | `schema apply` (declarative) | Declarative schema-as-code with a diff engine; supports *both* direct-apply (dev) and versioned migrations (prod) from one declared state. |
| **Django / Rails / Alembic** | autogenerated, versioned migrations | The **opposite** philosophy: autodetect model changes but write reviewable, reversible migration files. |

**What is genuinely different here** is the *packaging*, not the concept:
1. The sync is driven by an **external watch tool**, not the app process ‚Äî so the user's project stays
   free of any DB/migration coupling and the same model still compiles to the WASM frontend.
2. **Codegen (AST) instead of reflection** ‚Äî `ormc` already parses `model.go`, so the schema is known
   without a runtime model registry.
3. It's wired into a **hot-reload pipeline** (alongside browser reload / asset minify) for a
   real-time feel.

So: we're building a nicer-integrated `prisma db push` / `AutoMigrate`. That's a legitimate, useful
thing ‚Äî provided we're clear-eyed about the same trade-offs those tools carry.

## Advantages (honest)

- **Excellent early-dev DX.** Change a struct, the column appears. No migration ceremony, no
  context-switch. This is the right speed for greenfield iteration.
- **One source of truth in dev.** The Go struct. No drift between a model and a separate migration
  DSL, because there is no DSL.
- **Clean projects.** No sync boilerplate, no engine import in user code; the model stays isomorphic
  (backend + WASM). This is a real structural win over app-embedded auto-migrate.
- **Agnostic, testable core.** `orm` emits `Action`s; engines translate. The boundary is crisp and
  unit-testable without a DB.
- **Efficient.** The tool already parses `model.go` for codegen and reads each file once
  (`devwatch` + `depfind`); sync reuses that parse. No reflection, no extra walk.
- **Graceful degradation.** Without `TableIntrospector` the sync is purely additive (safe); with it,
  reconcile (rename/safe-drop) turns on. Engines opt in.

## Disadvantages & risks (honest)

1. **Dev/prod divergence ‚Äî the central weakness.** Additive push is *not* how production schema
   evolution works (review, reversibility, data backfills, zero-downtime). We now have **two
   evolution paths**: dev `Sync` and the deferred versioned migrations (PLAN ¬ß8). If they're not
   derived from the same source, "works in dev, breaks in prod" is the predictable failure. Every
   tool above that started with push-only (Prisma, GORM users) learned this. **The deferred part is
   the hard part.**
2. **"git is the history" is only half-true ‚Äî and the build log is not a substitute.** Git records
   the *model's* history, not the *DB's applied mutations*. ormc has its own logger in the build tab
   (app PLAN ¬ß3.3) and **must surface every action and error there ‚Äî nothing omitted**; that is
   essential for visibility. But that log is **ephemeral** (per-session, scrolls away, not
   machine-readable, not tied to DB state). It is *not* a **durable, replayable audit log**: you
   cannot reconstruct or roll back the DB's evolution from it, and additive sync is not invertible.
   So you can answer "what does the schema look like now?" but not "what changed, when, and can I roll
   it back?" The two are different properties ‚Äî visibility (have it) vs. reproducibility/rollback
   (don't).
3. **The dangerous no-ops must be loud, never silent.** Type changes are **ignored** (the column
   keeps its old type) and destructive changes are skipped. These must be **logged as visible
   warnings in the build tab** ‚Äî never absorbed silently ‚Äî or the developer assumes the DB matches
   the model when it doesn't. Likewise the additive log-and-continue fallback (used when an engine
   has no `TableIntrospector`) must **log every swallowed error**, not omit it. (With
   `TableIntrospector`, existing columns are skipped precisely, so swallowing is rare ‚Äî but when it
   happens it must still be visible.)
4. **Rename is a heuristic.** Without the `old_name` tag, a rename reads as drop+add. Safe-drop only
   drops *empty* columns, so a renamed **populated** column becomes "new empty column + orphaned old
   column" ‚Äî silent data stranding, not loss, but confusing.
5. **Dev DBs can drift between developers.** Because deletes-with-data and type changes don't apply, a
   long-lived dev DB is not guaranteed to equal a freshly-synced one. Two machines can hold different
   actual schemas from the same model. ("Just drop and re-sync" works, but that's a manual escape
   hatch, not a guarantee.)
6. **The tool mutates a real database on file save.** This is a footgun: if `DATABASE_CONNECTION`
   ever points at a shared/staging DB, every keystroke-save rewrites it. Needs a hard dev-only guard
   and a loud warning; "schema change triggered by a filesystem event" is inherently surprising.
7. **Dev engine may differ from prod engine.** The in-memory/local fallback is SQLite; prod is often
   Postgres. Schema that syncs cleanly on SQLite is validated against a *different dialect* than prod
   runs ‚Äî false confidence. (Mitigated only if the dev DSN points at Postgres too.)
8. **Introspection is chatty.** The safe-drop check runs `SELECT 1 ‚Ä¶ WHERE col IS NOT NULL LIMIT 1`
   per obsolete column, plus a `TableColumns` read per table, per sync. Dev-only and small, but it's
   N queries, not a single diff.
9. **Non-atomic across tables.** Per-model work can be wrapped in a Tx, but a multi-model sync isn't
   one transaction ‚Äî a mid-sync failure leaves a partially-evolved DB.
10. **Incremental-cache relation ordering.** Per-file processing resolves FK relations against a cache
    that fills as files arrive. During the startup walk a model referencing a not-yet-seen model may
    generate with incomplete relations until both are cached; dependents must be regenerated when a
    referenced model appears later, or the first pass is briefly inconsistent. (Correct by end of
    startup; a real edge to handle.)

## When this is the right call ‚Äî and when it isn't

**Right** for: greenfield/dev iteration, demos, in-memory/local prototyping, single-developer or
early-stage schemas that change hourly. The DX win is real and the safety cost is low when the data is
disposable.

**Wrong** as the *only* mechanism for: anything with production data, multi-developer schema
coordination, audit/rollback needs, or type/destructive changes. For those, push-mode must be paired
with a versioned, reviewable migration path derived from the *same* models ‚Äî which is exactly the
deferred PLAN ¬ß8.

## Recommendation

Ship the push-mode `Sync` (it's genuinely useful and the agnostic/codegen design is clean), but:
- **Hard-gate it to dev** (explicit flag/env, refuse non-dev DSNs, loud banner). Treat #6 as a
  first-class safety requirement, not a footnote.
- **Don't let ¬ß8 drift forever.** The honest risk is entrenching a push-only workflow that's painful
  to productionize. The credible end-state is Atlas/Ent/Prisma's: one declared schema ‚Üí *both* dev
  apply *and* versioned migration files. Plan the bridge early even if you build it later.
- **Surface the silent cases.** Log a visible warning on ignored type changes and on safe-drop
  skips/orphans, so "the DB didn't actually change" is never invisible.
- **Prefer the prod engine in dev** when available, so dev sync validates the dialect that ships.

Net: a well-built version of a proven dev-convenience pattern. The architecture (tool-driven,
agnostic, codegen, DB-blind `ormc`) is a nice improvement on app-embedded auto-migrate. The honest
caveat is the same one every push tool carries ‚Äî it is half of a schema story, and the deferred half
is the harder, more important one.
